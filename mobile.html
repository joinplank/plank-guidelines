01

Stack Choice & Architecture

Pick a framework deliberately, then layer it.

  • —Default to React Native + Expo when the team already ships React/TypeScript web (shared mental model, shared types with backend) — see the Frontend guide.
  • —Choose Flutter when the app is UI-heavy, needs pixel-identical cross-platform rendering, or the team has Dart depth.
  • —Layered architecture mirrors code-design.html: UI → state/view-model → service/repository → data source. No business logic in widgets/screens.
  • —RN feature-folder structure (features/<name>/{screens,components,hooks,api}), aligned with the Frontend co-location rule.
  • —Flutter: feature-first folders with presentation/domain/data per feature; avoid one giant lib/ of screens.
  • —One source of truth for shared types; RN shares Zod schemas/types with the backend where the monorepo allows.
02

State Management

Separate server state from client/UI state.

  • —RN: TanStack Query owns all server/async state — same rule as Frontend. No useEffect data fetching.
  • —RN client/UI state: Zustand (or Context for small trees); avoid Redux unless the client mandates it.
  • —Flutter: Riverpod as the default for app/async state; use AsyncValue for the loading/error/data tri-state.
  • —Keep ephemeral widget state local (useState / StatefulWidget); promote to a store only when shared.
  • —Never store server-cache state in a client store — caching and invalidation belong to the query layer.
03

Navigation

Typed, declarative, deep-link-ready routing.

  • —RN: Expo Router (file-based) or React Navigation with a typed param list — no untyped navigate('string').
  • —Flutter: go_router with typed routes; declare deep links centrally.
  • —Define the route tree in one place; screens receive params, never reach into global nav state.
  • —Specify and test deep linking and back-stack behavior per platform (Android hardware back, iOS swipe).
  • —Guard auth-gated routes at the router layer, not inside each screen.
04

Networking & Data

Typed client, validated boundaries.

  • —Single typed API client per app; parse responses with Zod (RN) / json_serializable + freezed (Flutter) at the boundary.
  • —Centralize base URL, auth headers, and retry/timeout policy — no per-call fetch config.
  • —RN: reuse backend contract types; align the error shape with the Backend guide's API conventions.
  • —Handle the offline/timeout case explicitly — surface a typed error, never a silent failure.
05

Native Modules & Platform APIs

Prefer managed, isolate native.

  • —RN: prefer Expo modules / config plugins before ejecting to bare native; document any native dependency.
  • —Wrap permissions (camera, location, notifications) behind a single typed abstraction; request lazily, handle denial.
  • —Flutter: prefer well-maintained pub.dev plugins; pin versions and audit platform-channel code.
  • —Keep platform-specific code behind Platform.OS / Platform.isIOS branches in one adapter, not scattered.
  • —Never block the UI thread on native bridge calls.
06

Performance & List Rendering

Lists and re-renders are the usual culprits.

  • —RN: use FlashList (or FlatList with keyExtractor, getItemLayout, windowing) for long lists — never .map() in a ScrollView.
  • —RN: memoize with React.memo/useCallback only at measured hotspots; avoid premature memoization.
  • —Flutter: use ListView.builder/slivers for long lists; prefer const constructors everywhere possible.
  • —Defer heavy work off the main thread; profile (Flipper / RN DevTools / Flutter DevTools) before optimizing.
  • —Optimize images: cached, correctly sized, lazy-loaded.
07

Offline & Local Storage

Choose storage by sensitivity and shape.

  • —Use secure storage (Keychain/Keystore) for tokens and secrets — never AsyncStorage/SharedPreferences for credentials.
  • —RN: MMKV or AsyncStorage for non-sensitive KV; a typed DB (WatermelonDB/SQLite) for relational/offline data.
  • —Flutter: flutter_secure_storage for secrets; Drift/Isar/shared_preferences by data shape.
  • —Define an explicit sync/conflict strategy for offline-first features; don't bolt it on later.
  • —Encrypt sensitive at-rest data; clear local caches on logout.
08

Accessibility

Ship accessible by default, like the web guide.

  • —Label every interactive element (accessibilityLabel / Semantics); icon-only buttons must have labels — mirrors Frontend Quality.
  • —Respect OS settings: dynamic font scaling, reduced motion, dark mode.
  • —Minimum touch target 44×44pt; ensure sufficient contrast against the token palette.
  • —Test with VoiceOver (iOS) and TalkBack (Android) on real flows.
09

Testing

Same pyramid as the Testing guide, mobile tooling.

  • —Follow the 70/20/10 pyramid — see testing.html. Unit-test view-models/services off-device.
  • —RN: Jest + React Native Testing Library for components/hooks; Maestro (or Detox) for E2E.
  • —Flutter: flutter_test widget tests; integration_test for E2E flows.
  • —Mock the network at the boundary (MSW-style for RN); never hit live backends in unit tests.
  • —Use snapshot tests sparingly — assert behavior, not pixel output.
10

Build, Release & CI

Reproducible builds, aligned with the CI/CD guide.

  • —RN: EAS Build for reproducible iOS/Android artifacts; OTA updates via EAS Update for JS-only changes.
  • —Flutter: build flavors for dev/staging/prod; produce signed release artifacts in CI, never on a laptop.
  • —Version and code-sign in CI; pull secrets from the CI vault, never commit them — see cicd.html.
  • —Automate store submission (Fastlane / EAS Submit); gate releases on tests + lint passing.
  • —Wire crash/error reporting (Sentry) before the first release — align with Infra observability.
