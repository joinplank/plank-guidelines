01

Strategy

Test behavior, not implementation. Coverage is a floor not a goal. Flaky tests are bugs — fix or delete immediately.

10%E2E
20%Integration
70%Unit
Unit Integration E2E
  • —Pyramid: many unit tests, fewer integration, minimal E2E. Ratio roughly 70/20/10.
  • —Test behavior, not implementation — no assertions on private methods or internal state.
  • —Coverage gate in CI (>80% lines), but coverage is a floor not a goal.
  • —Flaky tests are bugs — fix or delete immediately; never skip and move on.
02

Backend Tests

Real DB for integration tests, mocked repos for unit tests, contract tests on the HTTP surface.

  • —Unit: pure domain/use-case logic with mocked repositories.
  • —Integration: real DB (Postgres in Docker), real Prisma client, seeded fixtures.
  • —API contract tests against route handlers — test the HTTP surface, not internals.
  • —Property-based tests for data-transformation and parser functions (fast-check / Hypothesis).
03

Frontend Tests

Mock at MSW layer, not module level. Playwright for critical user flows only.

  • —Vitest for unit/component tests; React Testing Library for component behavior.
  • —Mock network at the MSW layer, not at module level — tests closer to real app behavior.
  • —Playwright for E2E critical paths only: auth, checkout, key user flows.
  • —Visual regression with Playwright screenshots on design-system components.
04

AI Pipeline Tests

Evals over unit tests for LLM outputs. Golden datasets, latency regression tests, and chaos injection for agent graphs.

  • —Evals over unit tests for LLM outputs — define expected quality metrics, not exact strings.
  • —Golden dataset of inputs/expected outputs; run evals on every prompt change in CI.
  • —Latency and cost regression tests: alert if avg tokens or p95 latency regresses >20%.
  • —Chaos tests for agent graphs: inject tool failures, verify state recovery and human escalation.
