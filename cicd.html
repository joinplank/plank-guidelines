01

Pipelines

Build once, promote the same image. Every step is idempotent. No secrets in images or committed files.

  • —Every PR runs lint, type-check, unit tests, and integration tests before merge — no exceptions.
  • —Build once, promote the same image artifact across environments (tag by Git SHA, not branch name).
  • —Secrets injected at runtime via environment-level secret stores — never baked into images or committed.
  • —Pipeline steps are idempotent: re-running a failed job must not cause side-effects.
02

Environments

Three tiers with automated promotion and a manual gate for production. Staging mirrors prod sizing exactly.

Tier 1
Dev
Auto-deploy on merge to main
→
Tier 2
Staging
Auto-deploy after dev passes
→
Tier 3
Production
Manual trigger + approval gate
  • —Three tiers: dev (auto-deploy on merge to main) · staging (auto-deploy after dev passes) · prod (manual trigger with approval gate).
  • —Environment-specific config via Helm values files — no branching logic inside pipeline YAML.
  • —Staging must mirror prod infra sizing and config; differences mask real failures.
  • —Feature flags over long-lived feature branches — ship dark, enable per environment.
03

Docker

Multi-stage builds, pinned base image digests, no secrets in build args, and CVE scanning in every run.

  • —Multi-stage builds: separate build and runtime stages. Final image from distroless or minimal base.
  • —Pin base image digests in production Dockerfiles, not floating tags.
  • —No secrets in build args or ENV instructions — use runtime secret mounts.
  • —Image vulnerability scan (Trivy) in CI; block merge on critical CVEs.
04

Deployments

Rolling deploys with readiness probes, automated rollback, migrations pre-deploy, and canary for high-risk changes.

  • —Rolling deploys with readiness probes gating traffic cutover — never replace all pods at once.
  • —Automated rollback on failed health checks within the first 5 minutes post-deploy.
  • —Database migrations run as a pre-deploy job, not inside application startup.
  • —Canary or blue/green for high-risk changes; document the strategy in the deploy PR.
05

Quality Gates

Use the checklist below for every deploy. All gates must pass — no silent failures.

  • —Coverage check (>80%) and bundle-size diff posted as PR comments, not silent failures.
  • —E2E smoke suite runs against staging after every deploy; page on-call if it fails.
  • —Dependency audit (npm audit / pip-audit) on every CI run — fail on high-severity advisories.
  • —Release notes auto-generated from conventional commits and attached to each GitHub release.
