What & why

The guidelines now cover mobile, and the SPA guide is complete

A reader picking a mobile stack — or filling a gap in our React SPA rules — had nowhere to look. Now they do.

  • There was no mobile guidance at all — engineers starting a Flutter or React Native app had no Plank defaults to follow.
  • The frontend SPA guide had only 4 sections; routing, state, forms, error boundaries, performance, and structure were missing.
  • Now: a new mobile.html guide (10 sections) + 6 new SPA sections, both in the existing house style.
The decision

One combined mobile page, not two

So you don't have to read the arch doc.

We chose

A single mobile.html covering both Flutter & React Native — they share ~60% of concerns (testing, CI, a11y) and readers compare them per-concern. Per-framework rules are prefixed RN:/Flutter: so search still filters them.

We rejected

Separate react-native.html + flutter.html — doubles the maintenance of cross-cutting rules and inflates the curated guide count for two stacks that overlap heavily.
What changed

The 3 things that matter

  • mobile.html (new) — 10 sections: stack choice, state, navigation, networking, native modules, performance, offline storage, a11y, testing, build/release. RN guidance cross-refs the frontend guide.
  • frontend.html:82,102+,177+ — appended SPA sections s05–s10 (routing, state, forms, error/suspense, performance, structure); s01–s04 byte-unchanged so anchors stay stable.
  • index.html:134,135 — new Mobile card (badges renumbered to 01–10); Guides 9→10; Sections stat corrected 34→57 (the 34 was stale).
