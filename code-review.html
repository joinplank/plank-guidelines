01

Overview

The /code-review skill reviews the current working diff for correctness bugs and reuse, simplification, and efficiency cleanups. Findings are structured and scoped to what actually changed.

# Review current diff at default effort
/code-review

# Specify effort level
/code-review low
/code-review high
/code-review ultra

# Review a GitHub PR by number
/code-review ultra 42

# Post findings as inline PR comments
/code-review --comment

# Apply findings directly to working tree
/code-review --fix
  • —Always run /code-review before opening a PR — it catches issues that tests can't surface, like logic bugs, dead code, and missed edge cases.
  • —The review operates on your working diff (uncommitted changes against the base branch), not the full file. Keep that diff focused.
  • —Requires a git repository. If you're not in one, run git init first.
02

Effort Levels

Five levels trade off speed against coverage. Default is medium when no level is specified. Higher levels cast a wider net but may include uncertain findings.

Level 1
low
High-confidence correctness bugs only. Fast — good for a quick sanity check before committing.
fast
Level 2
medium
Correctness bugs plus clear reuse and simplification opportunities. The default effort.
default
Level 3
high
Broader coverage — includes findings the model is less certain about. Good for pre-PR review of complex diffs.
thorough
Level 4
max
Exhaustive single-agent scan. May flag uncertain patterns. Best used on security-sensitive or high-risk changes.
exhaustive
Level 5
ultra
Deep multi-agent cloud review. Multiple reviewers collaborate and reconcile findings. Billed separately.
cloud
  • —Use low or medium for routine changes. Reserve high/max for diffs touching authentication, payments, or data migrations.
  • —Higher effort means more findings — not that every finding is actionable. Apply engineering judgment before acting on uncertain ones.
03

Flags

Two optional flags change what happens with the findings after the review completes.

--comment Post findings as inline comments on the GitHub PR. Requires a GitHub remote and an open PR for the current branch. Useful for async review workflows where teammates need to see findings in the PR interface.
--fix Apply findings directly to the working tree. Claude makes the edits it would recommend, so you can review the diff yourself before staging. Does not commit — you retain full control.
Flags combine with effort levels. For example: /code-review high --comment runs a high-effort review and posts the findings as PR comments.
  • —With --fix, always review the resulting diff before committing — the model edits what it's confident about, but you own the final code.
  • —With --comment, findings appear as line-level PR comments. Resolve or dismiss them before merging.
04

Ultra Review

Ultra launches a multi-agent cloud review. Multiple independent agents analyse the diff and reconcile their findings, producing higher-confidence results than a single-pass review.

# Review the current branch (bundles local diff — no GitHub remote needed)
/code-review ultra

# Review a specific GitHub PR
/code-review ultra 42

# Review with inline PR comments posted
/code-review ultra --comment
  • —Ultra is user-triggered and billed separately — it cannot be launched automatically by Claude or from a hook.
  • —The no-argument form (/code-review ultra) bundles the local branch and does not require a GitHub remote.
  • —Use ultra for PRs that are large, touch multiple subsystems, or require a second opinion before merging to main.
  • —/ultrareview is a deprecated alias for /code-review ultra — prefer the new form.
05

Simplify

/simplify is a focused companion to /code-review. It looks only at quality — reuse, simplification, efficiency, and altitude — and applies fixes directly. It does not hunt for bugs.

CommandFinds bugsQuality cleanupsAuto-applies fixes
/code-reviewYesYesWith --fix
/simplifyNoYes — primary focusAlways
  • —Run /simplify after /code-review --fix to catch cleanup opportunities the bug-focused pass may have missed.
  • —Do not rely on /simplify alone for correctness — it intentionally skips bug detection.
06

Recommended Workflow

Integrate code review into your existing spec-driven workflow. The review step sits between implementation and the PR, not after it.

StageCommandWhen
During implementation/code-review lowQuick correctness check after completing a subtask
Before opening a PR/code-review highFull review of the complete diff; apply or address findings
After applying fixes/simplifyQuality pass to reduce noise and consolidate code
For complex or high-risk PRs/code-review ultraMulti-agent deep review before merging to main
Async team review/code-review --commentPost findings directly on GitHub so teammates see them inline
Worktree tip: when using the Writer/Reviewer pattern, open a second Claude session in a fresh worktree and run /code-review high there. Because that session never wrote the code, its review is unbiased.
  • —Code review and tests serve different purposes — tests verify correctness against specs; review surfaces design issues, dead code, and subtle bugs that tests don't cover. Run both.
  • —Address or explicitly dismiss every high-confidence finding before merging. Leaving findings open without a comment creates ambiguity for future readers.
  • —For security-sensitive diffs (auth, payments, RLS policies), use /code-review max or ultra — the extra coverage is worth the time.
