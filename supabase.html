01

Persistence

Treat Supabase as a managed PostgreSQL host — migrations in version control, pooled connections, and typed access at every layer.

Connection string selection
# Transactional workloads (APIs, serverless) → use the pooler
DATABASE_URL=postgresql://postgres.[ref]:[pass]@aws-0-[region].pooler.supabase.com:6543/postgres?pgbouncer=true

# Long-lived processes (migrations, cron jobs) → direct connection
DIRECT_URL=postgresql://postgres.[ref]:[pass]@aws-0-[region].pooler.supabase.com:5432/postgres
prisma/schema.prisma — dual URL setup
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")    // pooler for runtime
  directUrl = env("DIRECT_URL")      // direct for migrations
}
  • —Use the Supavisor pooler (port 6543, transaction mode) for all runtime database connections — never connect directly from serverless or short-lived processes.
  • —Keep a DIRECT_URL (port 5432) exclusively for prisma migrate deploy and other schema operations that require a persistent session.
  • —All schema changes go through migration files (prisma migrate dev / supabase db diff) committed to version control — never run raw DDL in production or use db push.
  • —Wrap multi-step writes in transactions; partial state on failure is never acceptable.
  • —Add indexes intentionally — profile with EXPLAIN ANALYZE before and after; rely on the supabase-postgres-best-practices skill for query optimization rules.
  • —Prefer timestamptz over timestamp for all datetime columns; store everything in UTC.
  • —Use uuid primary keys (default in Supabase) — never auto-increment integers exposed in public APIs.
02

API

Supabase exposes PostgREST, Realtime, and Storage APIs. Use the typed client on the frontend; keep sensitive logic server-side or in Edge Functions.

lib/supabase/server.ts — server-side client
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'
import type { Database } from '@/lib/supabase/database.types'

export function createClient() {
  const cookieStore = cookies()
  return createServerClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { getAll: () => cookieStore.getAll(), setAll: (c) => c.forEach(({ name, value, options }) => cookieStore.set(name, value, options)) } }
  )
}
  • —Generate TypeScript types from the schema: supabase gen types typescript --linked > lib/supabase/database.types.ts — commit and regenerate on every migration.
  • —Use @supabase/ssr (not the legacy auth-helpers) for all Next.js / SvelteKit server-side clients; it handles cookie-based session refresh correctly.
  • —Never expose the service_role key to the client or any public endpoint — it bypasses RLS entirely.
  • —Prefer PostgREST column selection (.select('id, name')) over .select('*') — over-fetching is a security and performance risk.
  • —Use Realtime only for truly push-driven UX (notifications, presence, live scoreboards) — polling with useQuery is simpler and cheaper for most features.
  • —Enable Realtime per-table in the Supabase dashboard; do not broadcast tables containing PII or sensitive data.
  • —For Storage: set bucket policies to private by default; create signed URLs server-side with short expiry (expiresIn: 60) for sensitive files.
  • —Validate file MIME type and size server-side before writing to Storage — client-side validation is advisory only.
03

Row Level Security Required

RLS is the primary data-isolation boundary in Supabase. Every table accessed by the anon or authenticated role must have RLS enabled with explicit policies — no exceptions.

Minimum RLS setup for a user-owned table
-- 1. Enable RLS and force it even for table owner
alter table posts enable row level security;
alter table posts force row level security;

-- 2. Users can only read their own rows
create policy "posts: owner read"
  on posts for select
  to authenticated
  using (user_id = auth.uid());

-- 3. Users can only insert rows they own
create policy "posts: owner insert"
  on posts for insert
  to authenticated
  with check (user_id = auth.uid());

-- 4. Users can only update/delete their own rows
create policy "posts: owner modify"
  on posts for update, delete
  to authenticated
  using (user_id = auth.uid());
Avoid N+1 in policies — use security definer functions
-- Bad: subquery re-evaluated per row
create policy "team member read"
  on documents for select
  using (team_id in (select team_id from team_members where user_id = auth.uid()));

-- Good: stable function evaluated once
create or replace function auth.user_team_ids()
returns setof uuid language sql stable security definer as $$
  select team_id from team_members where user_id = auth.uid();
$$;

create policy "team member read"
  on documents for select
  using (team_id = any(array(select auth.user_team_ids())));
  • —Enable RLS on every table that the anon or authenticated role can reach — including junction tables and lookup tables with sensitive keys.
  • —Use force row level security so that even the table owner (Postgres superuser excluded) cannot bypass policies.
  • —Write separate policies per operation (select, insert, update, delete) — overly broad for all policies hide intent and complicate audits.
  • —Always use auth.uid() (not a passed parameter or header) as the identity anchor — it cannot be spoofed by a client.
  • —Wrap repeated subqueries in security definer Postgres functions marked stable — the planner will cache them per query instead of re-evaluating per row.
  • —Audit RLS coverage regularly: query pg_tables joined against pg_policies to find tables with RLS disabled or zero policies.
  • —Test policies in isolation — write integration tests that authenticate as different users and assert that cross-tenant data is inaccessible.
04

Authentication

Supabase Auth issues JWTs consumed directly by PostgREST and RLS policies. Keep tokens server-side, rotate secrets, and guard every protected route.

middleware.ts — Next.js session refresh
import { createServerClient } from '@supabase/ssr'
import { NextResponse, type NextRequest } from 'next/server'

export async function middleware(request: NextRequest) {
  let response = NextResponse.next({ request })
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll: () => request.cookies.getAll(),
        setAll: (cookiesToSet) => cookiesToSet.forEach(({ name, value, options }) =>
          response.cookies.set(name, value, options)
        ),
      },
    }
  )
  // Refresh session — must call getUser() not getSession()
  await supabase.auth.getUser()
  return response
}

export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
}
  • —Call supabase.auth.getUser() (not getSession()) on the server to validate the JWT against Supabase Auth on every request — getSession() reads the cookie without server-side verification.
  • —Run the session-refresh middleware on every route that might set cookies; missing it causes auth state to go stale after the JWT expires (default 1 hour).
  • —Protect routes server-side — redirect unauthenticated users in middleware or Server Components, never rely solely on client-side guards.
  • —Use OAuth providers (Google, GitHub, etc.) or Magic Link for consumer apps; reserve email/password only when required by the client's spec.
  • —Store custom user data (display name, role, preferences) in a profiles table linked to auth.users via a trigger — never write directly into auth.users.
  • —Implement role-based access by storing roles in app_metadata (server-set, not editable by users) and checking (auth.jwt() ->> 'app_metadata')::jsonb ->> 'role' inside RLS policies.
  • —Set JWT expiry to 3600 s (1 hour) in the Supabase dashboard and enable Refresh Token Rotation — never issue non-expiring tokens.
  • —Rotate the JWT secret on a schedule or immediately after a suspected compromise — all existing sessions are invalidated on rotation.
05

Edge Functions

Edge Functions run Deno at the edge — use them for webhooks, scheduled jobs, and server-side logic that must not live in the client. Keep functions small and stateless.

supabase/functions/send-welcome/index.ts
import { createClient } from 'jsr:@supabase/supabase-js@2' const corsHeaders = { 'Access-Control-Allow-Origin': Deno.env.get('ALLOWED_ORIGIN') ?? '', 'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type', } Deno.serve(async (req) => { if (req.method === 'OPTIONS') return new Response('ok', { headers: corsHeaders }) const supabase = createClient( Deno.env.get('SUPABASE_URL')!, Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')! // server-side only ) const { userId } = await req.json() // ... business logic return new Response(JSON.stringify({ ok: true }), { headers: { ...corsHeaders, 'Content-Type': 'application/json' }, }) })
supabase/functions/.env — local secrets (never commit)
ALLOWED_ORIGIN=https://app.example.com RESEND_API_KEY=re_…
  • —Always handle OPTIONS preflight requests and return the correct CORS headers — missing this is the most common Edge Function debugging trap.
  • —Scope CORS: set Access-Control-Allow-Origin to the specific domain, never * when the function uses auth credentials or sensitive data.
  • —Store secrets with supabase secrets set KEY=value (or via the dashboard) — never hard-code secrets or commit .env files.
  • —Use the service_role key only inside Edge Functions (server environment) — it is never passed to the client or stored in env vars prefixed with NEXT_PUBLIC_.
  • —Verify the caller's JWT when an Edge Function is invoked by a user: extract Authorization: Bearer <token> and call supabase.auth.getUser(token) before processing.
  • —For webhook receivers (Stripe, Resend, GitHub) verify the provider's signature before trusting the payload — reject unsigned requests with 403.
  • —Keep functions single-purpose and stateless — shared logic belongs in a shared module imported by multiple functions, not in a mega-function with branching paths.
  • —Use supabase functions serve locally with a .env file for development — test against a local Supabase instance (supabase start) to avoid hitting production data.
