01

Component Design

Server Components by default in the App Router. Push the client boundary as far toward the leaves as possible.

Pattern

Server-first

Default to Server Components. Add 'use client' only when you need interactivity, hooks, or browser APIs.

Pattern

Props down, events up

Avoid prop drilling beyond 2 levels. Use context or a state manager for shared state across the tree.

Pattern

Shadcn customization

Shadcn components are copied source. Customize via cn() and variants — never edit the primitive directly.

Pattern

Co-location

Keep component, styles, and tests in the same folder. One directory = one responsibility.

  • —Server Components by default in Next.js App Router — client boundary only when needed (interactivity, hooks, browser APIs).
  • —Shadcn components are copied source — customize via cn() and variants, not direct edits to primitives.
  • —Props down, events up. Avoid prop drilling beyond 2 levels — use context or state manager.
  • —Co-locate component, styles, and tests in the same folder.
02

Data Fetching

TanStack Query owns all client-side async state. No exceptions for useEffect-based fetching.

  • —TanStack Query for all client-side async state. No useEffect for data fetching.
  • —Define query keys as typed constants, not inline strings.
  • —Optimistic updates for mutations on user-facing actions.
  • —Server Actions for form mutations in App Router; avoid REST round-trips for simple writes.
03

Styling

Tailwind utilities only. Design tokens live in tailwind.config, not scattered as hardcoded values.

  • —Tailwind utility classes only — no custom CSS files unless for global resets or Tailwind theme tokens.
  • —Design tokens via tailwind.config theme extension, not hardcoded values in classnames.
  • —Responsive-first: mobile breakpoint is the base, md:/lg: for wider screens.
  • —Dark mode via Tailwind dark: variant — never manual class toggling.
04

Quality

Strict TypeScript, Zod at boundaries, accessibility by default, and bundle hygiene before merging large deps.

  • —Strict TypeScript: match backend strictness config. No as any.
  • —Zod for form validation and API response parsing on the client.
  • —Accessibility: semantic HTML, ARIA labels on icon-only buttons, keyboard navigation tested.
  • —Bundle analysis before merging large deps — check with Next.js bundle analyzer.
05

Routing

TanStack Router owns navigation; typed end to end.

  • —TanStack Router for client routing with fully typed routes, params, and search params — no untyped string paths.
  • —Co-locate route definitions with their feature; loaders fetch via TanStack Query, not ad-hoc effects.
  • —Validate search params with Zod; treat the URL as typed state.
  • —Guard authenticated routes at the router/loader layer, not inside components.
06

State Management

Server state ≠ client state.

  • —Server state lives in TanStack Query (see Data Fetching). Client/UI state in Zustand or Context — never duplicate server data into a client store.
  • —Keep state as local as possible; lift only when genuinely shared.
  • —Derive, don't store: compute from existing state instead of syncing copies.
  • —No global mutable singletons; stores are typed and scoped.
07

Forms & Validation

One schema, reused everywhere.

  • —React Hook Form for form state; Zod resolver for validation — a single schema shared with the API contract.
  • —Validate on blur/submit, not on every keystroke; surface field-level errors accessibly.
  • —Disable submit while pending; show optimistic or pending UI for mutations.
  • —Never trust client validation alone — the server re-validates (see the Backend guide).
08

Error & Suspense Boundaries

Failures and loading are first-class UI.

  • —Wrap route segments in error boundaries; render a recoverable fallback, not a white screen.
  • —Use Suspense + TanStack Query useSuspenseQuery for declarative loading states.
  • —Report caught errors to Sentry with feature context — align with Infra observability.
  • —Distinguish expected errors (validation, 404) from unexpected (crash) in the UI.
09

Performance

Ship less JS, render less often.

  • —Code-split by route; lazy-load heavy/below-the-fold components with dynamic / React.lazy.
  • —Memoize (useMemo/useCallback/React.memo) only at measured hotspots — profile first.
  • —Optimize images with next/image; prefer Server Components to shrink client bundles.
  • —Run the bundle analyzer before merging large dependencies.
10

Project Structure

Predictable, feature-first layout.

  • —Feature-first folders: features/<name>/{components,hooks,api,schemas} — co-locate by feature.
  • —Shared primitives in components/ui (Shadcn); shared logic in lib.
  • —Centralize design tokens in tailwind.config — no scattered magic values.
  • —Expose one public surface per feature; avoid deep cross-feature imports.
